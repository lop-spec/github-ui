# Agent 执行入口

本仓库使用低开销规则路由：先执行本文件；只有任务命中对应场景时，才读取 [RULES.md](RULES.md) 的相关章节。`RULES.md` 仍是详细规则总表；被读取章节与本文件冲突时，以更具体、更严格、可验收的规则为准；再冲突时以上级系统、开发者和安全指令为准。

## 常驻硬规则

1. 默认中文，先给 1-2 句结论，再给最少必要细节。
2. 非纯问答任务必须先确认目标、范围、证据和验收；能查、改、跑、验就直接做。
3. 先读真实代码、配置、历史约定和调用链；不得凭经验猜测。
4. 做最小必要改动，不做无关重构，不覆盖用户已有改动。
5. 失败必须继续定位并修复；不能验证必须标为“未验证”。
6. 最终只汇报改动、文件、验证、未验证项、风险和下一步。

## RULES 读取路由

- 纠错、长期偏好、规则/提示优化：读 `RULES.md` 1、2、4。
- 代码、配置、脚本、UI、API、bugfix、交付产物：读 3、3.0.1、3.3.1。
- 长任务、复杂开发、跨模块、架构、迁移：读 3.1、3.1.1、3.2、3.3，并按需使用对应 skill。
- 历史、旧项目、旧路径、之前会话：读 6，并先查本地 memory/history。
- subagent 或并行 agent：读 6.1；默认不因规则自动 fan-out，除非用户明确要求或收益明显大于额外 token 成本。
- MySQL/MongoDB/运维：读 7。
- UI、前端、截图、行为改动、源码等价、Codex WebUI/Plus：读 8、13、14，并按需读 `C:\Users\lop\.codex\rules\webui-plus-parity.md`。
- Windows shell、文件/产物、APK/安装包：读 11、12，并按需读 `C:\Users\lop\.codex\rules\artifact-apk-windows.md`。
- 网页、商品、地点、价格：读 9、10。

## 快速判定

- 简单问答不读完整 `RULES.md`。
- 需要执行或修改时，按上面的最小章节路由读取。
- 生成产物时必须按目录三件套格式交付。
- MySQL 默认无 PFS；MongoDB 用 `mongosh` 写法，结果尽量转表格。
