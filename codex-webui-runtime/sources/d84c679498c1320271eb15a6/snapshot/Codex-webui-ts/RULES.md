# Codex WebUI 项目规则

## 当前范围

当 lop 指定 Codex WebUI/Plus 基本 UI 与功能还原范围时，必须按从左到右优先级执行：

1. 对话按钮功能
2. 对话框显示格式
3. 对话框支持的超链接格式
4. 左侧导航栏显示
5. 对话栏显示
6. 项目分类
7. 设置页
8. 设置选项
9. 其他基本功能

## 强制边界

- 基本 UI 与功能还原必须覆盖会话/发送/停止/历史/错误等基础可用链路。
- 禁止把截图或 Plus 源码里的高级设置项扩成当前任务必做范围，除非 lop 明确要求全仓库、全源码、所有 Plus 模块或所有高级功能。
- 独立 App 前台入口必须默认使用 Chrome，不得自动调用 Edge；桌面/开始菜单/前台窗口入口必须使用项目自绘图标，不能退化成浏览器图标。
- 当 lop 点名账户、导航栏账户信息、对话框、speed、计划模式、完全访问权限、+ 号菜单、模型/权限/发送按钮等具体 Plus 功能块，或说“这部分一对一/完全还原/按源码逻辑”时，该命名块必须按 Plus 源码逻辑、UI、字段、入口、状态、错误态和可见信息做 source-to-target ledger 与逐项验收；不得用“基本 UI 范围”“高级功能/设置页不在范围”或旧记忆中的 basic scope 排除、降级或转移到设置页。
- 当 lop 点名 timeline、工具卡、file change/fileChange、diff/patch 卡片或默认展开/折叠/隐藏状态时，默认可见状态、summary 文案、状态徽标、详情内容和 toggle/disclosure 都属于 Plus 源码等价验收；file change/diff/patch 外层详情默认收起，除非 Plus 源码或 lop 明确要求展开。交付前必须用静态测试或运行态 DOM 断言 `details.open`、`aria-expanded`、`hidden` 等状态，禁止只断言类名、文本或卡片数量。
- 左侧导航栏项目分类已确认后，顶层全局“最近”会话区不再是兼容入口或回退方案，项目区内部也不得展示“最近项目”分组或用“最近”命名折叠/空状态。会话入口必须收敛到项目分组内；置顶、运行中、搜索结果等例外必须有独立语义和验收证据，禁止继续命名为“最近”。交付前必须检索源码/DOM/截图，确认未保留未被明确允许的“最近”入口或“最近项目”文案。
- 当 lop 指出 WebUI 功能/UI 缺陷并同时要求追根因、改规则或改提示时，必须先修复并验证当前 WebUI 缺陷，再把导致漏做的执行规则固化；禁止只改规则后结束。
- 禁止假按钮、固定提示词、假数据或视觉近似冒充完成。
- 临时下线或隐藏 UI 入口时，禁止用 HTML 注释块保留可见布局或留下注释闭合符；必须删除入口 DOM/事件绑定残留，并用静态测试和运行态页面检查确认无残留文本。
- 修改 WebUI 字体、字号、背景、主题色变量或全局可读性样式前，必须先备份受影响 CSS 和规则文件；最终汇报必须列出备份文件名和验证结果。
- 每次交付前必须运行受影响链路的构建、测试和浏览器验证；不能验证的项必须标为未验证。
- 为部署、接口、SSE、UI、回归验证、subagent 验证或运行态探测创建的临时测试会话，必须使用可识别测试标记；验证结束前必须通过真实归档或隐藏接口清理，并二次查询确认不再出现在 lop 的真实会话/对话列表里。右键删除或清理会话必须按会话日期移动到 `Codex_RECYCLE` 冷归档并从列表隐藏，不得硬删。若清理失败，必须先修复清理路径并重试，禁止把测试会话遗留给 lop 手动清。

## Codex App Plus 工程规则映射

本节从 `C:\Users\lop\Documents\Codex\Windows\codex-app-plus-1.8.1-source\AGENTS.md` 映射而来。执行 WebUI 任务时必须按 WebUI 的 Node/TypeScript/SSE 架构理解这些规则；禁止把 Plus 的 React/Tauri/Rust 规则原样套成不适用要求。

### 项目快照与命令

- 本项目是 Windows 本地优先的 Codex WebUI，服务端为 Node.js 18+、TypeScript、内置 HTTP/SSE，前端为 `public/` 下的原生 HTML/CSS/JS。
- 运行命令以 `package.json` 为准：构建用 `npm run build`，静态语法检查用 `npm run check`，测试用 `npm test`，本地服务用 `npm run serve`，公网/隧道相关用 `npm run public` 或现有启动脚本。
- 迭代时先运行最窄相关检查；交付前按影响范围运行更宽的 `npm run build`、`npm run check`、`npm test` 和必要的运行态接口/UI 验证。

### 目录与模块边界

- 服务端入口和 HTTP 路由在 `src/server.ts`；Codex 子进程、JSON/SSE 事件和会话生命周期在 `src/services/codex.ts`；文件、会话扫描、路径校验等通用逻辑在 `src/utils/`。
- 前端交互逻辑放在 `public/js/`，样式放在 `public/css/`，`public/index.html` 只保留必要结构和挂载点；禁止把新逻辑继续堆进单个巨型 HTML/JS 文件。
- 测试放在 `tests/`，优先补受影响端点、会话扫描、配置、SSE 和 UI 可验证链路。
- `dist/` 是构建产物，除非任务明确要求修产物或验证构建输出，否则禁止手改。

### 代码风格

- 匹配现有模块边界和命名；优先在负责该功能的 service、utils、public/js 或 public/css 附近增加小函数。
- TypeScript 必须保持严格、清晰和可维护；避免松散 `any`、未使用变量和隐式协议形状。
- 优先复用现有配置解析、路径校验、会话扫描、Codex JSON/SSE、广播和认证辅助函数，不重复造一套并行实现。
- 注释保持少而有用，只解释非显然的生命周期、协议、Windows 路径、并发、锁或兼容性决策。

### 前后端契约

- 新增或修改能力必须更新完整链路：服务端路由/类型/校验、前端请求与状态处理、必要样式、测试和文档/规则。
- 变更配置项时必须同时覆盖默认值、白名单/校验、持久化读写、使用点、UI 控件和测试。
- 变更 SSE 或 Codex 事件时必须同时覆盖服务端事件解析、广播 payload、前端订阅处理、错误/断连/重连状态和测试或运行态验证。
- 涉及文件系统、会话删除、导入导出、上传下载或项目 cwd 的功能必须做路径归一化、越权路径防护和 Windows 路径验证；会话删除的目标必须限制在 `Codex_RECYCLE` 日期桶内。

### 测试与验证

- reducer、解析器、路径校验、会话扫描、配置读写优先写小而确定的单元测试。
- HTTP 路由和 WebUI 行为优先补 Node test；涉及真实页面时必须用运行态接口、DOM/控制台/截图或等价证据验证。
- 所有运行态探测创建的临时会话必须带可识别测试标记并清理确认。

### 生成文件、工作区与外部资料

- `dist/`、临时输出、上传/传输目录和历史备份不得作为源文件直接维护；修改源文件后重新构建或说明为何只读验证。
- 工作区可能有用户或并行修改，禁止回滚或覆盖无关改动；涉及提交/暂存时必须显式路径，禁止 `git add .`。
- 需要当前库、框架、SDK、CLI 或云服务行为时，优先查官方文档；纯项目内业务逻辑、重构或排障不得用外部文档替代真实代码。
